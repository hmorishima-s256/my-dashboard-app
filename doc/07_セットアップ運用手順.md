# 07. セットアップ運用手順

## 7.1 前提

- Node.js / npm が利用可能
- Google CloudでOAuthクライアント作成済み
- Google OAuth クレデンシャル（`credentials.json`）をローカルに配置済み

推奨配置（リポジトリ外）:

- `~/my-dashboard-app/_shared/credentials.json`
- Windows 例: `C:\\Users\\<ユーザー名>\\my-dashboard-app\\_shared\\credentials.json`

補足:

- `MY_DASHBOARD_CREDENTIALS_PATH` 環境変数で明示パス指定も可能
- 互換維持のため、プロジェクトルートなどの既存探索もフォールバックで有効

## 7.2 初期セットアップ

```bash
npm install
```

## 7.2.1 Dev Container を使う場合（推奨）

前提:

- WSL2
- Rancher Desktop（`Container Engine: dockerd (moby)`）
- VS Code
- VS Code拡張 `Dev Containers`

手順:

1. VS Codeでプロジェクトを開く
2. コマンドパレットで `Dev Containers: Reopen in Container` を実行
3. コンテナ起動後、`postCreateCommand` で `npm install` が自動実行される
4. ターミナルで `npm run dev` を実行

Electron画面の表示:

- VS Codeのポート転送で `6080`（Desktop/noVNC）をブラウザで開く
- コンテナ内デスクトップ上にElectronウィンドウが表示される
- Dockerfile に Electron ランタイム依存（`xvfb` 等）がプリインストール済みのため追加設定不要

注意:

- Dev ContainerはLinux環境のため、`npm run build:win` はWindowsホスト側で実行する

## 7.2.2 Dev Container 再構築時の保持仕様

Dev Container では named volume を利用し、以下を再構築後も保持する。

- `node_modules`
- GitHub CLI 認証情報（`/root/.config/gh`）
- Codex 作業データ（`/root/.codex`）
- VS Code Remote 拡張キャッシュ（`/root/.vscode-server/extensions`）
- VS Code Remote ユーザーデータ（`/root/.vscode-server/data/User`）

初回のみ必要な認証:

```bash
gh auth login
```

補足:

- `gh` は Dev Container イメージにプリインストール済み（手動 `apt install` は不要）
- トークン期限切れ時は再認証が必要
- volume を削除した場合は、拡張機能キャッシュや認証情報も再作成となる

## 7.2.3 Dev Container の標準拡張機能セット

標準拡張機能は `.devcontainer/devcontainer.json` の
`customizations.vscode.extensions` で定義する。

標準拡張（13件）:

- `dbaeumer.vscode-eslint`
- `esbenp.prettier-vscode`
- `github.vscode-pull-request-github`
- `github.vscode-github-actions`
- `redhat.vscode-yaml`
- `davidanson.vscode-markdownlint`
- `yzhang.markdown-all-in-one`
- `streetsidesoftware.code-spell-checker`
- `christian-kohler.path-intellisense`
- `EditorConfig.EditorConfig`
- `vitest.explorer`
- `openai.chatgpt`
- `mhutchie.git-graph`

運用方針:

- Build/Rebuild 時は、上記拡張が Dev Container 側へ自動適用される。
- 1 回目の適用はネットワーク状態により時間がかかる場合がある。
- 拡張キャッシュ volume が保持されている場合、再構築時の再取得は基本不要。
- volume 削除時は拡張を再取得する。

## 7.2.4 Build/Rebuild 後の拡張適用確認手順（チェックリスト）

### A. `customizations.vscode.extensions` 定義拡張の適用確認

1. `Dev Containers: Rebuild Container` を実行する。
2. コンテナ起動後、ターミナルで次を実行する。

```bash
required=(
  dbaeumer.vscode-eslint
  esbenp.prettier-vscode
  github.vscode-pull-request-github
  github.vscode-github-actions
  redhat.vscode-yaml
  davidanson.vscode-markdownlint
  yzhang.markdown-all-in-one
  streetsidesoftware.code-spell-checker
  christian-kohler.path-intellisense
  EditorConfig.EditorConfig
  vitest.explorer
  openai.chatgpt
  mhutchie.git-graph
)

missing=()
for ext in "${required[@]}"; do
  if ! ls /root/.vscode-server/extensions | grep -q "^${ext}-"; then
    missing+=("$ext")
  fi
done

if [ "${#missing[@]}" -eq 0 ]; then
  echo "OK: 標準拡張13件を確認"
else
  echo "NG: 不足拡張"
  printf ' - %s\n' "${missing[@]}"
fi
```

3. `OK: 標準拡張13件を確認` が出れば合格。

### B. volume保持時の再取得不要確認

1. volumeを削除せずに再度 `Dev Containers: Rebuild Container` を実行する。
2. Aの確認コマンドを再実行し、同様に `OK` を確認する。
3. 大量の拡張ダウンロードが再発していないことを確認する。

### C. volume削除時の再取得確認

1. Dev Container関連volumeを削除する。
2. `Dev Containers: Rebuild Container` を実行する。
3. 初回同様に拡張が再取得されることを確認する。
4. Aの確認コマンドで `OK` を確認する。

補足:

- volume削除で再作成される対象には、拡張キャッシュ・GitHub CLI認証情報・Codexデータが含まれる。
- トークン期限切れ時やvolume削除時は、`gh auth login` 等の再認証が必要。

## 7.2.5 機密ファイル運用方針

- 機密ファイルはリポジトリに置かない（コミットしない）。
- 標準運用は `~/my-dashboard-app/_shared/` 配下へ集約する。
- `.env` は現時点で必須ではない（必要時のみローカルで作成）。
- `.env` を使う場合、共有用は `.env.example` のみを管理し、実値は `.env*` でローカル保持する。

非追跡確認コマンド:

```bash
git status --short --ignored
git ls-files | rg -n "credentials\\.json|token\\.json|client_secret|service_account|\\.env" -S
```

## 7.2.6 AI 開発環境（Claude Code + Electron Skill）

本プロジェクトでは Claude Code と Electron Skill（agent-browser）を開発補助ツールとして採用している。

### Claude Code

- `CLAUDE.md`（プロジェクトルート）に振る舞い・ルール・開発フローを集約
- `.claude/settings.json` でツール権限を管理

### Electron Skill (agent-browser)

Chrome DevTools Protocol（CDP）経由で Electron アプリを自動操作するツール。

インストール済み:

- コマンド: `agent-browser`（`/usr/local/share/npm-global/bin/agent-browser`）
- バージョン: `0.16.x`
- Skill 定義: `.claude/skills/electron/SKILL.md`（`.agents/skills/electron/SKILL.md` へのシンボリックリンク）
- ロック: `skills-lock.json`

開発時の動作確認フロー:

```bash
# デバッグポート付きで起動
npm run dev -- --remote-debugging-port=9222

# 別ターミナルで接続・確認
agent-browser connect 9222
agent-browser snapshot -i          # UI要素一覧
agent-browser screenshot check.png # スクリーンショット
```

### なぜこの構成か

- ターミナル完結の開発フローを維持しつつ、UI の動作確認を自動化できるため
- Electron は CDP ポートをネイティブに持つため、追加ツール不要で自動操作できるため

## 7.3.1 開発フロー（個人運用）

```bash
# 1. developmentから作業ブランチを切る
git switch development && git pull --ff-only
git switch -c feat/xxxx

# 2. 実装

# 3. DoD チェック（マージ前に全パス必須）
npm run format && npm run lint && npm run typecheck && npm run test

# 4. PR 作成 → CI 確認 → マージ
gh pr create --base development --fill
gh pr checks --watch
gh pr merge --squash --delete-branch
git switch development && git pull --ff-only
```

ブランチ命名規則: `feat/*` / `fix/*` / `chore/*` / `docs/*`

`main` へのマージは機能・ドキュメントが揃ったタイミングで手動判断する。

## 7.4 ビルド

共通ビルド:

```bash
npm run build
```

OS別:

```bash
npm run build:win
npm run build:mac
npm run build:linux
```

Windowsでインストーラ不要の場合:

```bash
npm run build:unpack
```

### 補足（なぜこの運用か）

- `build:win` は署名編集で環境依存エラーが起きやすいため、設定で `signAndEditExecutable: false` を採用
- ローカル検証を優先する場合は `build:unpack` の方が安定しやすい

## 7.5 初回ログイン運用

1. アプリ起動
2. 右上 `Googleログイン`
3. ブラウザで認証/同意
4. ブラウザに完了ページ表示
5. アプリが前面に戻り、当日予定が同期される

## 7.6 日常運用

- 予定同期: 右上 `同期`
- 日付移動:
  - 日付右の `◀` / `▶` で前日/翌日
  - または日付入力パネルで直接指定
- タスク:
  - `タスク登録` で追加
  - 一覧から `詳細` で更新
  - `開始/中断/再開/停止/完了` で進捗更新
- 設定:
  - 左下歯車 -> 時刻/間隔/時間表示を設定 -> 保存

## 7.7 常駐と終了

- ウィンドウの `×` で確認ダイアログ
  - `バックグラウンド常駐`: トレイへ格納
  - `終了`: 完全終了

### なぜこの仕様か

- ユーザーの意図（最小化したいのか終了したいのか）が分かれるため、毎回明示的に選べるようにしている

## 7.8 Windows自動起動運用

- 条件:
  - Windows
  - パッケージ版
- 初回1回だけ `openAtLogin=true` を設定
- 再設定防止:
  - `_shared/auto-launch-initialized`
- 隠し起動:
  - `--hidden`

## 7.9 トラブルシュート

### ログインできない

- `credentials.json` の `redirect_uris` に `localhost` があるか確認
- `~/my-dashboard-app/_shared/credentials.json` または `MY_DASHBOARD_CREDENTIALS_PATH` の指定先にファイルがあるか確認
- OAuth同意画面の設定を確認

### 予定が取得できない

- ログイン状態を確認
- 対象日付に予定があるか確認
- 該当カレンダーの閲覧権限を確認

### 自動取得が動かない

- 設定保存済みか確認
- 時刻形式が `HH:mm` か確認
- 間隔が1以上か確認
- 未ログインでは実行されない点を確認

### `npm run build:win` で失敗する

- キャッシュ権限エラー時は以下を試す
  - ターミナルを管理者で実行
  - `%LOCALAPPDATA%\electron-builder\Cache` のクリア
- まずは `npm run build:unpack` で動作確認する
